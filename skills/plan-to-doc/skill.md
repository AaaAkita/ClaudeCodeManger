# 计划文档化 Skill

## 描述

当用户完成一项技术方案或实施计划的讨论后，主动将该计划整理成结构清晰的 Markdown 文档，写入项目目录中。帮助团队沉淀设计决策、保留上下文，并便于后续执行与回顾。

## 触发条件

当用户明确表达以下意图时触发：
- "把计划写成一个文档"
- "把这个计划记录下来"
- "帮我整理成文档"
- "写个文档保存到项目里"
- "把这个方案文档化"

## 工作流程

### 1. 收集计划内容
- 从当前会话中提取已确认的技术方案、实施步骤、阶段划分
- 回顾用户此前明确接受的设计决策和关键约束

### 2. 结构化文档
文档应包含以下章节：
- **背景与目标**：为什么要做这件事，解决什么问题
- **技术选型**：关键技术的对比与选择理由
- **现状分析**：已有的基础和缺失的能力
- **实现路径**：分阶段的详细步骤，每阶段的目标、前后端工作、交互效果
- **冲突解决/风险控制策略**：如何处理边界情况
- **关键文件清单**：明确的新建文件和修改文件列表
- **验证方式**：如何测试和验收

### 3. 确定存储位置
- 优先保存到项目目录下的 `docs/` 文件夹
- 若 `docs/` 不存在，可创建该目录或保存到项目根目录
- 文件名应清晰反映计划主题，如 `XXX实现计划.md`

### 4. 写入文档
- 使用 `Write` 工具将内容写入指定路径
- 文档使用标准 Markdown 格式，包含适当的标题层级、表格和代码块

### 5. 反馈用户
- 告知用户文档已保存的完整路径
- 简要说明文档结构和后续可执行的操作

## 输出标准

- 文档逻辑清晰，便于未参与讨论的人也能快速理解
- 包含足够的技术细节（接口、表结构、文件路径），可直接作为开发依据
- 阶段划分明确，每阶段有独立的验收标准
- 使用中文撰写（除非项目主要使用英文）

## 注意事项

- 不要遗漏用户已经确认的关键决策（如选型理由、取舍）
- 不要在文档中加入尚未确定的假设内容
- 保存文档后，可主动询问用户是否需要立即进入实施
